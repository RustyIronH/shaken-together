  <action>
Rewrite `src/main.ts` to be the application entry point that wires all modules together:

```typescript
// 1. Initialize
const canvas = document.getElementById('game-canvas') as HTMLCanvasElement;
const uiRoot = document.getElementById('ui-root') as HTMLDivElement;

// 2. Size canvas to viewport
resizeCanvas(canvas);
window.addEventListener('resize', () => {
  resizeCanvas(canvas);
  // TODO: Reposition boundaries on resize if needed
});

// 3. Create physics engine
const engine = createPhysicsEngine();

// 4. Create scene (boundaries + default ragdolls)
const scene = createScene(engine, canvas.width, canvas.height);

// 5. Set up debug renderer
const { ctx } = createDebugRenderer(canvas);

// 6. Set up multi-touch drag
const cleanupTouch = setupMultiTouch(canvas, engine, scene);

// 7. Create UI controls
const panel = createPanel(uiRoot, {
  onDollCountChange: (count) => {
    setDollCount(engine, scene, count, canvas.width, canvas.height);
  },
  onModeChange: (mode) => {
    applyMode(engine, scene, mode);
  },
  onReset: () => {
    resetScene(engine, scene, canvas.width, canvas.height);
  },
});

const hamburger = createHamburger(uiRoot, () => panel.toggle());

// 8. Start physics engine
const runner = startEngine(engine);

// 9. Render loop (separate from physics -- physics runs via Runner)
function gameLoop() {
  renderFrame(ctx, engine, scene);
  requestAnimationFrame(gameLoop);
}
requestAnimationFrame(gameLoop);
```

Key integration details:
- Physics runs via Matter.Runner (fixed timestep, decoupled from render)
- Rendering runs via requestAnimationFrame (synced to display refresh)
- These are intentionally separate: physics is deterministic, rendering is best-effort
- Multi-touch events go through the drag system which modifies physics constraints
- UI panel callbacks directly call world management functions
- Canvas resize updates canvas pixel dimensions (not just CSS) to prevent coordinate mismatch

Remove any remaining Vite boilerplate (counter, CSS imports to deleted files, etc.).
