   ```
     Get its RagdollSprite from spriteMap
     For each [label, body] in ragdoll.bodies:

       Set graphic.position.set(body.position.x, body.position.y)
       Set graphic.rotation = body.angle
   ```
   Remove the old `requestAnimationFrame(gameLoop)` loop entirely -- PixiJS Ticker replaces it.

3. **Drag feedback** -- when a ragdoll is being dragged, scale its container to 1.05:

     - Build a `Set<string>` of currently dragged ragdoll IDs from `scene.activeDrags`

4. **Z-order via PixiJS** -- update `src/renderer/colors.ts`:
   - Keep `bringToFront` for physics composite ordering (still needed for drag-manager's composite splice/push)
   - Add a new export `bringContainerToFront(ragdollLayer: Container, ragdollContainer: Container): void` that does `ragdollLayer.removeChild(ragdollContainer); ragdollLayer.addChild(ragdollContainer);`
   - In drag-manager's `handlePointerDown`, after the existing composite z-order splice, also call the PixiJS container reorder. To avoid coupling drag-manager to PixiJS, use a callback pattern:

     - In main.ts, provide an `onDragStart` callback that calls `bringContainerToFront` for the dragged ragdoll's sprite container
     - Alternatively, handle z-order in the ticker based on activeDrags (simpler): in the ticker, the last-dragged ragdoll's container gets moved to the end of ragdollLayer children. This avoids modifying drag-manager at all.
   - **Recommended approach:** Handle z-order entirely in the ticker callback in main.ts. Check `scene.activeDrags` and ensure dragged ragdolls' containers are at the end of ragdollLayer children. This keeps the drag system untouched.

5. **Update screenToWorld in drag-manager.ts** for PixiJS DPR handling:
   - The existing `screenToWorld` uses `canvas.width / rect.width` ratio. With PixiJS `autoDensity: true`, the canvas CSS size matches the window but pixel size = window * devicePixelRatio. The ratio `canvas.width / rect.width` will equal devicePixelRatio, which is correct. However, PixiJS stage coordinates are in CSS pixels (not canvas pixels) when autoDensity is true. So we need to use the CSS coordinate directly:
   - Update `screenToWorld` to: `x = clientX - rect.left`, `y = clientY - rect.top` (no scaling). With autoDensity, PixiJS maps CSS pixels to world coordinates automatically. BUT Matter.js bodies are positioned in pixel coordinates matching the old canvas (width = window.innerWidth). With PixiJS `autoDensity`, `app.screen.width` = window.innerWidth (CSS pixels), so physics world coordinates match CSS coordinates. The ratio still works.
   - Actually, with autoDensity=true, `canvas.width = innerWidth * dpr` but `rect.width = innerWidth`, so ratio = dpr. The physics world was created with `canvas.width = innerWidth` (old canvas), but now we pass `app.screen.width` which is also innerWidth. So screenToWorld ratio = dpr, meaning world coords = css coords * dpr. This would be wrong.
   - **Correct approach:** Do NOT change screenToWorld. Instead, when creating the scene, pass `app.screen.width` and `app.screen.height` (which are CSS pixel dimensions with autoDensity). The physics world uses these CSS-pixel dimensions. screenToWorld's ratio (canvas.width/rect.width = dpr) is wrong in this case. Fix: change screenToWorld to NOT scale (just subtract rect offset) since physics coordinates now match CSS coordinates. OR: pass `app.screen.width` and `app.screen.height` as the canvas "width" and "height" for hit testing.
   - **Simplest fix:** Don't use `canvas.width` in screenToWorld -- use `rect.width` for both dimensions, making the scale factor 1.0. Update the function to:
     ```typescript
     return {
       x: clientX - rect.left,
       y: clientY - rect.top,
     };
     ```
     This works because PixiJS autoDensity means stage coordinates = CSS coordinates = physics coordinates.

6. **Sync spriteMap on doll count changes and reset:**

     - For `onReset`: No sprite changes needed -- positions are updated automatically by the ticker sync.
   - Extract this into a helper function `syncSprites(scene, spriteMap, ragdollLayer)` inside main.ts.

   **Anti-patterns to avoid:**
   - Do NOT clear and rebuild Graphics every frame -- only update position/rotation/scale
   - Do NOT use requestAnimationFrame -- use app.ticker.add()
   - Do NOT forget to wrap in async IIFE for Vite compatibility (Research Pitfall 3)

1. Create `src/renderer/effects/squash-stretch.ts`:
   - Export `applySquashStretch(graphic: Graphics, body: Body): void`
   - Calculate speed from `body.velocity`: `speed = sqrt(vx^2 + vy^2)`
   - Max deformation = 0.2 (20% per user decision)
   - `deformFactor = min(speed / 15, 0.2)`
   - Stretch along movement direction by setting `graphic.scale.set(1 - deformFactor * 0.5, 1 + deformFactor)`
   - When speed < 0.5, reset to `scale.set(1, 1)` (no jitter at rest)

2. Create `src/renderer/effects/speed-lines.ts`:
   - Export `SpeedLinePool` interface and `createSpeedLinePool(effectsLayer: Container, maxLines: number): SpeedLinePool`
   - Export `updateSpeedLines(pool: SpeedLinePool, ragdolls: RagdollInstance[], spriteMap: Map<string, RagdollSprite>): void`
   - Speed lines are short Graphics lines (length ~10-20px) that trail behind fast-moving body parts
   - Velocity threshold: only show when body speed > 8
   - Each line: a thin Graphics rect (2px wide, 10-15px long) with alpha 0.4, positioned behind the body part (opposite direction of velocity), rotated to match velocity angle
   - Use an object pool of ~20 Graphics objects. Each frame: deactivate all, then activate needed ones at new positions. Objects not needed get `visible = false`.
   - Line color: white with alpha 0.3-0.5
   - Only check a subset of body parts for performance: head, upperTorso, handL, handR, footL, footR (6 per ragdoll, not all 15)

3. Create `src/renderer/effects/impact-flash.ts`:
   - Export `ImpactFlashPool` interface and `createImpactFlashPool(effectsLayer: Container, poolSize: number): ImpactFlashPool`
   - Export `spawnImpactFlash(pool: ImpactFlashPool, x: number, y: number, force: number): void`
   - Export `updateImpactFlashes(pool: ImpactFlashPool, deltaMs: number): void`
   - Each flash is a Graphics star burst (4-6 pointed star shape using polygon points) colored white
   - Pool of ~8 flash Graphics objects, initially `visible = false`
   - `spawnImpactFlash`: finds an inactive flash, positions it at (x, y), sets scale proportional to `force` (min 0.5, max 2.0), sets `visible = true`, marks active with remaining lifetime = 150ms
   - `updateImpactFlashes`: each frame, decrement lifetime by delta. Reduce alpha linearly. When lifetime <= 0, set `visible = false`.
   - Only spawn for impacts where `force > 3` (relative velocity threshold from Research)
   - Force = `sqrt((bodyA.velocity.x - bodyB.velocity.x)^2 + (bodyA.velocity.y - bodyB.velocity.y)^2)` (NOT pair.normalImpulse which is unreliable -- Research Pitfall 6)

4. **Wire effects into main.ts:**
   - Import all 3 effect modules
   - Create pools: `const flashPool = createImpactFlashPool(renderer.effectsLayer, 8)` and `const speedLinePool = createSpeedLinePool(renderer.effectsLayer, 20)`
   - Set up collision event handler using `Events.on(engine, 'collisionStart', ...)`:
     - For each pair, calculate impact force from relative velocity
     - If force > 3: `spawnImpactFlash(pool, midpointX, midpointY, force)`
     - If force > 5: change face to 'surprised' on both ragdolls' head Graphics (swap context to surprised expression for ~800ms, then back to neutral; if force > 10, use 'dazed' for ~1500ms)
   - In the ticker callback, after position sync:
     - Call `applySquashStretch(graphic, body)` for each body part during the position sync loop
     - Call `updateSpeedLines(speedLinePool, scene.ragdolls, spriteMap)`
     - Call `updateImpactFlashes(flashPool, ticker.deltaMS)`

5. **Face expression state management** (in main.ts or a small helper):
   - Track face state per ragdoll: `Map<string, { expression: FaceExpression, timer: number }>`
   - On collision: set expression to 'surprised' or 'dazed' with a timer
   - In ticker: decrement timer by deltaMS. When timer <= 0, swap head context back to 'neutral'
   - Use `getCharacter(ragdoll.characterId).faceExpressions.get(expression)` to get the correct GraphicsContext
   - Apply via `headGraphic.context = newContext`

   **Performance notes:**
   - All effect objects are pre-created and pooled -- no allocations during gameplay
   - Speed lines check only 6 body parts per ragdoll, not all 15
   - Impact flashes have a max pool of 8 -- excess collisions are silently dropped
   - Squash-stretch is just a scale transform -- zero allocation cost
