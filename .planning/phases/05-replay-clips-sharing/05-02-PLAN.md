  <files>src/capture/share.ts, src/capture/__tests__/share.test.ts, src/ui/toast.ts</files>
  <behavior>
    - shareFile(blob, filename, mimeType) with Web Share API available and canShare=true: calls navigator.share with File object, returns 'shared'
    - shareFile when navigator.share throws AbortError (user cancelled): returns 'shared' (not an error)
    - shareFile when Web Share API unavailable and mimeType='image/png': copies to clipboard via ClipboardItem, returns 'copied'
    - shareFile when Web Share API unavailable and mimeType='image/gif': triggers download via anchor element, returns 'downloaded'
    - shareFile when all methods fail: returns 'failed'
    - dataUrlToBlob(dataUrl) converts a data URL string to a Blob
    - showToast(container, message, durationMs) creates a fixed-position pill notification that fades in, auto-dismisses after durationMs, and removes itself from DOM after fade
  </behavior>
  <action>
    Create src/capture/share.ts exporting:

    - `type ShareResult = 'shared' | 'copied' | 'downloaded' | 'failed'`
    - `async function shareFile(blob: Blob, filename: string, mimeType: string): Promise<ShareResult>` -- implements the fallback chain from research:
      1. Try Web Share API: create File from blob, check navigator.share && navigator.canShare?.({ files: [file] }), call navigator.share({ files: [file], title: 'Shaken Together' }). Return 'shared'. Catch AbortError -> return 'shared' (user cancelled is not failure). Other errors -> fall through.
      2. Clipboard fallback (PNG only): if mimeType === 'image/png' && navigator.clipboard?.write, create ClipboardItem({ 'image/png': blob }), return 'copied'. Catch -> fall through.
      3. Download fallback: create object URL from blob, create anchor with download=filename, click, revoke URL. Return 'downloaded'.
      4. If somehow all fail: return 'failed'.
    - `async function dataUrlToBlob(dataUrl: string): Promise<Blob>` -- uses fetch(dataUrl).then(r => r.blob()).

    Create src/ui/toast.ts exporting:
    - `function showToast(container: HTMLElement, message: string, durationMs?: number): void` -- default duration 2000ms. Follow the onboarding-hint pattern exactly: create div, inline styles (position: fixed, bottom: 80px, left: 50%, transform: translateX(-50%), background: rgba(0,0,0,0.8), color: #fff, padding: 10px 20px, borderRadius: 20px, fontSize: 14px, zIndex: 2100, opacity: 0, transition: opacity 0.2s ease, pointerEvents: none). Append to container. requestAnimationFrame -> opacity '1'. setTimeout(durationMs) -> opacity '0' -> transitionend with once:true removes from DOM.

       - **Effects toggle** -- "Effects: OFF"/"Effects: ON" pill button, only visible (display: flex/none) when Photo tab is active. Toggles between clean and withEffects preview images.
       - **Share button** -- blue (#2196F3) pill button with text "Share".        - **Save button** -- green (#4CAF50) pill button, text "Save". Same content logic as Share but calls callbacks.onSave(blob, filename). Fade out overlay before calling callback.
       - **Discard button** -- transparent pill with border, text "Discard". Fades out overlay then calls callbacks.onDiscard(). Same pattern as current.

    **Key behaviors:**
    - Default tab: "Photo" (matches current behavior)
    - Effects toggle state resets to OFF on each overlay open
    - Share/Save on Photo tab: filename = `shaken-together-{timestamp}.png`, mimeType = 'image/png'
    - Share/Save on Clip tab: filename = `shaken-together-{timestamp}.gif`, mimeType = 'image/gif'
    - If GIF is not yet ready when user taps Share/Save on Clip tab, show toast "GIF encoding..." and do nothing else
    - Save button fades out overlay before callback (same as current). Share button does NOT fade overlay (user stays on overlay after sharing).
    - Discard fades out overlay then calls onDiscard (same as current).
    - Import shareFile from '../capture/share' and dataUrlToBlob from '../capture/share'. Import showToast from './toast'.
    - The Share button handler calls callbacks.onShare which in main.ts will call shareFile and show toast based on result.

    Actually, to keep the overlay self-contained and avoid main.ts needing to handle share results in a callback: the Share button should directly call shareFile and showToast internally. This avoids an awkward callback-within-callback pattern.

    **Revised approach for Share button:** Import shareFile and dataUrlToBlob from share.ts, and showToast from toast.ts directly in capture-overlay.ts. The Share button:
    1. Gets current blob (Photo: await dataUrlToBlob(currentImgSrc), Clip: stored gifBlob)
    2. Calls await shareFile(blob, filename, mimeType)
    3. If result is 'copied', showToast(container, 'Copied to clipboard!')
    4. If result is 'downloaded', showToast(container, 'Saved to device!')
    5. If result is 'failed', showToast(container, 'Could not share')

    This means we simplify the callbacks:
    ```typescript
    export interface CaptureOverlayCallbacks {
      onSave: (blob: Blob, filename: string) => void;
      onDiscard: () => void;
    }
    ```
    No onShare callback needed -- share logic lives in the overlay.

    **Save button handler:**
    1. Gets current blob same as Share
    2. Creates object URL, creates anchor with download attribute, clicks, revokes URL
    3. Fades overlay out, then calls callbacks.onSave(blob, filename) for resume logic (physics resume happens here)

    Wait, actually the onSave callback currently triggers physics resume. The new save should:
    1. Download the file
    2. Fade overlay
    3. Call onDiscard() (which resumes physics) -- or keep onSave for the resume signal

    Simplest: callbacks = { onDismiss: () => void } -- one callback that resumes physics. Save and Share handle their own file operations. Discard just dismisses.

    **Final callback interface:**
    ```typescript
    export interface CaptureOverlayCallbacks {
      onDismiss: () => void;  // Called after overlay is removed -- resume physics + re-enable capture button
    }
    ```

    - Save button: downloads file, fades overlay, calls onDismiss
    - Discard button: fades overlay, calls onDismiss
    - Share button: shares file (stays on overlay -- does NOT dismiss)

    This is cleaner and matches the user decision that Save downloads and Share opens the share sheet.
