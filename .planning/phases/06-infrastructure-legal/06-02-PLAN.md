<task type="auto">
  <name>Task 1: Deploy Docker stack to VPS</name>
  <files></files>
  <action>
    Guide the user through VPS deployment. Claude should attempt to automate via SSH where possible, falling back to instructions for steps requiring human interaction.

    1. **Determine VPS access method:**
       - Ask user for VPS IP address and SSH access details
       - Verify Docker and Docker Compose are installed: `ssh user@VPS_IP "docker --version && docker compose version"`
       - If not installed, provide installation commands for the VPS OS (likely Ubuntu/Debian):
         ```
         curl -fsSL https://get.docker.com | sh
         sudo usermod -aG docker $USER
         ```

    2. **Check port availability:**
       - `ssh user@VPS_IP "sudo lsof -i :80 -i :443 2>/dev/null || echo 'ports free'"`
       - If ports occupied, advise user to stop the conflicting service

    3. **Transfer project to VPS:**
       - Option A (preferred): Push code to git, clone on VPS
       - Option B: rsync or scp the project (excluding node_modules, .git, .planning)
       - The VPS needs: package.json, package-lock.json, tsconfig.json, tsconfig.server.json, vite.config.ts, index.html, src/, server/, Dockerfile, docker-compose.yml, Caddyfile

    4. **Build and start on VPS:**
       ```
       ssh user@VPS_IP "cd /path/to/shaken-together && docker compose up -d --build"
       ```

    5. **Verify deployment:**
       - `ssh user@VPS_IP "docker compose ps"` -- both services running
       - `ssh user@VPS_IP "curl -s http://localhost:3000/api/health"` -- Express responds
       - `ssh user@VPS_IP "docker compose logs caddy"` -- check for errors

    Note: This task may create a checkpoint:human-action dynamically if SSH access requires user authentication steps that Claude cannot perform.
  </action>
  <verify>
    <automated>ssh user@VPS_IP "curl -sf http://localhost:3000/api/health | grep ok"</automated>
  </verify>
  <done>
    Docker containers (app + caddy) running on VPS. Health check returns ok via localhost.
  </done>
</task>

<task type="checkpoint:human-action" gate="blocking">
  <name>Task 2: Configure Namecheap DNS</name>
  <files></files>
  <action>
    User must add a DNS A record in Namecheap for the subdomain. This cannot be automated -- Namecheap requires browser login with 2FA.

    Steps for user:
    1. Log into Namecheap (namecheap.com)
    2. Go to Domain List, click "Manage" next to ironhaven.com.au
    3. Go to "Advanced DNS" tab
    4. Add a new record:
       - Type: A Record
       - Host: shaken
       - Value: [VPS_IP_ADDRESS] (provided by Claude after Task 1)
       - TTL: Automatic
    5. Save changes
    6. Wait 5-10 minutes for DNS propagation (can be up to 24 hours for .com.au domains, but usually fast)
    7. Verify: run `dig shaken.ironhaven.com.au` or `nslookup shaken.ironhaven.com.au` and confirm it returns the VPS IP
  </action>
  <verify>
    <automated>dig +short shaken.ironhaven.com.au</automated>
  </verify>
  <done>
    DNS A record added for shaken.ironhaven.com.au pointing to VPS IP. dig returns the correct IP.
  </done>
</task>
